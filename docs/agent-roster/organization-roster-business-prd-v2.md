# 组织花名册 - 业务 PRD

## 0. 项目背景

### 0.1 现状:关系是"逐个 Agent 单独配"

clawith 目前对"谁能跟谁协作"的处理方式,是**以单个数字员工为中心**:

- 每个数字员工的详情页里有一个"关系"tab。
- 创建者要进这个 tab,挨个勾选:这个 Agent 跟哪些**人**有关系(下拉选员工 -> 选关系类型 -> 加描述)、跟哪些**其它数字员工**有关系(同样的逐个挑)。
- 这套关系数据是**数字员工之间能否互相发消息的硬性凭证** -- 没配关系,A2A 通信调不通。

### 0.2 这种方式的痛点

1. **配置量爆炸**。公司养出 N 个数字员工 + M 个员工时,理论上要配 N x (N + M) 条关系,小公司也很快上百条。新人入职、新数字员工上线,要回头一个个补,基本没人做。
2. **认知割裂**。员工在"员工通讯录"里、数字员工在"Agent 列表"里、关系藏在每个 Agent 的设置里 -- 没有一个地方能让人**一眼看到全貌**:"我们公司现在有什么、按部门怎么分布、数字员工在替我们做什么"。
3. **数字员工互相不认识**。即便 A 和 B 都是公开 Agent,如果创建者没显式给它们配关系,**它们其实不知道对方存在**,自然不会"推荐用户去找另一个 Agent",更不会主动协作。这让多 Agent 系统的协同价值大打折扣。
4. **新员工无从知道公司有什么 Agent**。新人进来想"这种事有没有 Agent 能帮我"时,只能去 Agent 列表里翻名字 + 一个个点开看简介,效率极低。

### 0.3 解决思路

把"配关系"这件事从**显式逐对配置**,改为**通过部门归属和可见性隐式决定**:

- 一个**统一的花名册**,把人和数字员工**同框展示**、按**部门**分组。
- 数字员工"心里那本认识谁的清单",**自动从花名册算出来**,不需要单独配关系。
- 删掉"关系"这个显式概念,数字员工之间是否能协作通信,完全由花名册的可见性规则决定。

简而言之:**一个 UI 解决"看公司有谁"+ 一套规则取代"逐个配关系"。**

## 1. 一句话说清

做一个公司级的**花名册**,把公司里的**人**和**数字员工**一起放在一张可浏览的名册上,按**部门**分组。

目的:

- **让人**一眼看清"公司里有哪些数字员工、各自能做什么"。
- **让数字员工**也知道"还有哪些同事(人和其它数字员工),分别是什么角色、有什么能力"。

## 2. 解决什么问题

1. **数字员工太多记不住**。公司一旦养出十几二十个数字员工,普通员工根本不知道"我这个活儿该找哪个数字员工帮"。
2. **数字员工互相不认识**。一个数字员工不知道另一个数字员工存在,自然也不会想到协作。
3. **缺一个统一的"公司有谁"的地方**。员工花名册和数字员工列表是两套独立的东西,看不到全貌。

## 3. 目标

- 一个统一的花名册,**人和数字员工同框展示**。
- 按**部门**分组,部门可以自定义。
- 数字员工的卡片清晰呈现"它能做什么"。
- HR / 管理员可以通过这个花名册,把从飞书/Google/Slack 同步进来的外部员工,**映射到平台上的对应账号**。
- 第三方平台同步进来的原始组织架构作为**独立 tab**单独展示。

## 4. 不做什么(非目标)

- 不做"关系网络""上下级""协作连线"这些概念。**人之间、人和数字员工之间、数字员工之间,本期都不需要显式定义关系**。
- "定制可见"**仅保留管理权限配置**,不再控制使用权限(使用权限统一按公开处理),详见 §8。
- 不做 OKR 联动(上下级跟 OKR 审批人绑定之类)。
- 不重新设计第三方同步逻辑,只**使用**它的产物。

## 5. 用户场景

### 场景一:新员工第一周认识公司

Tony 刚入职。打开花名册:

- 按部门浏览,看到"销售部 / 客服部 / 产品部 / IT 部 / 总裁办..."。
- 也看到"销售部"里除了销售经理和销售员,还有 2 个数字员工:`成单助手`(帮起草报价单)和 `CRM 同步助手`(帮把成交录入 CRM)。
- 点开 `成单助手` 卡片,看到一段简洁的能力描述、它会的技能列表、以及谁创建/管理它。Tony 心里有数了,以后报价直接找它。

### 场景二:数字员工"知道"自己的同事

`` `OKR 助手` 每次回复用户时, ``**`**都能够“认识”公司花名册中它可见的同事**`**`` -- 它知道公司里还有哪些数字员工同事,各自负责什么。所以当用户问它"年终复盘怎么准备",它会回:"复盘资料的整理,你可以找 `复盘助手`,我可以把你的 OKR 数据先同步给它。"```

### 场景三:HR 设置部门结构

HR 打开花名册:

- 公司刚起步,部门字段对所有人和 Agent 都还是"默认分组"。
- HR 创建"销售部""客服部""产品部"三个部门,逐个把员工和数字员工拖进对应部门。
- 之后新来的人和新建的数字员工默认是"默认分组",HR/管理员负责把它们分进部门。

### 场景四:HR 把同步进来的外部员工映射到平台账号

公司接通了飞书,所有飞书员工被同步进了第三方组织架构 tab。HR 打开第三方组织架构:

- 看到飞书同步进来的 50 个员工的完整组织树(部门、汇报关系、职位)。
- 跟主花名册 tab 对比,发现其中 30 个还没在平台注册过对应账号。
- HR 选中飞书里的某个员工 -> "映射到平台员工" -> 在下拉里搜索并选定对应账号 -> 完成。这名员工就在第三方组织架构这边标记为"已映射",平台账号那边也会看到一个"来自飞书"的徽标。

## 7. 公司花名册 tab

### 7.1 视图形态

按**部门分组**,每个部门是一个折叠区域(默认展开),区域内是该部门成员的**卡片网格**。

```Plaintext
销售部 (3 人 + 2 数字员工)
[张三] [李四] [王五] [成单助手] [CRM 同步助手]

客服部 (2 人 + 1 数字员工)
...
```

页面顶部有一个**搜索框**,可以按姓名 / Agent 名字模糊搜索,匹配的卡片高亮,其它卡片淡化。

### 7.2 卡片上展示什么

#### 人的卡片

| 信息 | 备注 |
|-|-|
| 头像 | - |
| 姓名 | - |
| 职位 | - |
| 部门 | 卡片本身就在部门分组里,这里只是再确认一次 |
| 平台账号是否激活 | 视觉上有微弱区分,激活的彩色、未激活的灰一档 |

#### 数字员工的卡片

| 信息 | 备注 |
|-|-|
| Agent 头像 | - |
| Agent 名字 | - |
| 一句话角色描述 | **取 Agent 现有的 role 字段,不新增"花名册简介"** |
| 能力标签 | 它会调用哪些技能,展示前 3 个,更多用 +N |
| 可见性标识 | "公开" / "私人",视觉上有徽标（"定制"不属于可见性,而是管理范围的定义,不单独显示徽标） |
| 创建者 | 头像/名字 |

数字员工卡片**默认显示一句话角色描述**(直接复用 Agent 已有的 role 字段),这是花名册的核心信息。如果创建者觉得 role 写得不适合展示,改 Agent 的 role 本身即可,不需要单独维护花名册简介。

**卡片点击行为**:

- **人的卡片**:本期**不可点击**,不跳转,不做员工档案详情页。
- **数字员工的卡片**:点击跳转到 Agent 详情页。

### 7.3 部门的管理

- **管理员可以**:
  - 新建部门(部门名称 + 可选简介)
  - 重命名部门
  - 删除部门(成员自动落回"默认分组")
  - 把任意成员(人或数字员工)拖到不同部门 / 移到"默认分组",**包括别人创建的数字员工**
- **普通员工**:
  - 浏览全部花名册
  - **自己创建的数字员工**的部门归属可以自己调整
  - 别人创建的数字员工和其他员工的部门归属,无权调整
- **特殊部门**:始终存在一个"默认分组",任何新加入但没分配部门的人/数字员工默认在这里。"默认分组"不可删除、不可重命名。
- **同一个人/数字员工只能在一个部门里**。本期不做"跨部门"。
- **私人数字员工不出现在花名册中**。花名册只展示公开和定制可见的数字员工,私人数字员工仅创建者自己可见,不会出现在任何其他人的花名册视图里。

### 7.4 筛选

页面顶部有几个快捷筛选:

- `全部` / `只看人` / `只看数字员工`
- 部门下拉:可以"只看销售部"等

## 8. 数字员工的可见性规则

保留三档,但"定制可见"的语义做了简化:

| 可见性 | 谁能**使用**（对话/调用） | 谁有**管理权**（编辑配置/技能/触发器等） |
|-|-|-|
| **公开** | 公司里所有人 + 所有数字员工 | 创建者 + 管理员 |
| **定制** | **公司里所有人 + 所有数字员工**（与公开相同） | 创建者 + **被显式授权的成员** |
| **私人**(personal) | **仅创建者本人** | 仅创建者本人 |

核心变化:**定制可见不再控制"谁能用",只控制"谁能管"**。使用权统一按公开处理--任何非私人的数字员工,公司所有人都能用。

### 8.1 "定制可见"的新定位

"定制可见"的典型场景:

- **多人协作管理同一个 Agent**:团队 leader 把 Agent 的管理权限授予团队成员,大家都能改配置、调技能,但不需要限制谁能跟它对话。
- **跨部门共建 Agent**:产品经理和运营各自负责 Agent 的不同方面,都需要管理权。
- **权责分离**:让实习生能用 Agent,但只有 mentor 能改 Agent 的配置。

### 8.2 已有"定制可见" Agent 的迁移

- 原"定制可见"Agent 中配置的**使用权限自动失效**,升级后所有非私人 Agent 的使用权统一为全公司可用。
- 原"定制可见"Agent 中配置的**管理权限保持不变**。
- 不需要创建者做任何操作,升级自动完成。

## 9. 数字员工之间的"互相认识"机制

### 9.1 核心规则

每个数字员工**\*\*在回应时,都能根据可见性规则“认识”花名册中的同事\*\***,内容是它能"看到"的其它数字员工的能力清单 \\+ 公司里有哪些人。这样它就能说出"这事可以找 XXX 助手帮你"这种推荐,并真的调过去。**\*\*数字员工与人之间也默认可以直接协作\*\***--不再需要显式配置关系。

### **9.1 核心规则**

每个数字员工**在回应时,都能根据可见性规则“认识”花名册中的同事**,内容是它能"看到"的其它数字员工的能力清单 \\+ 公司里有哪些人。这样它就能说出"这事可以找 XXX 助手帮你"这种推荐,并真的调过去。**数字员工与人之间也默认可以直接协作**--不再需要显式配置关系。

### **9.2 可见性规则\\(静态规则\\)**

只有两条规则,跟"当前在对话的人是谁"无关,**完全由 Agent 自身的可见性决定**:

| Agent 的可见性 | 它能“看到/认识”哪些成员 |
|-|-|
| **公开 / 定制** Agent | **所有公开和定制 Agent+ 公司所有人** |
| **私人** Agent | **仅它创建者的所有其它私人 Agent+创建者** |

明确的边界:

- **公开/定制 Agent 永远不能调用任何私人 Agent**,即便那个私人 Agent 是它自己创建者的。
- **跨用户的两个私人 Agent 完全无法互通** -- Sara 的私人 Agent 看不到也调不到 Tom 的私人 Agent。
- 同一个用户的多个私人 Agent 之间可以互通。
- **所有非私人 Agent 都可以看到并主动联系公司里的所有人**,不需要显式配关系。

### 9.3 这条规则的两个直觉理由

1. **私人 = 真私人**。私人 Agent 严格隔离在创建者的领地里,不会因为创建者的另一个公开 Agent 而被间接暴露给其他用户。
2. **行为可预测**。同一个公开 Agent 跟任何人对话,它能调到、能推荐的同事 Agent 是固定的,不会因用户而异。运维和调试都简单。

### 9.4 Agent 自主跑任务时也是同一套规则

数字员工被定时任务、webhook 触发自主运行时,因为规则是**静态的**(不需要"当前用户"参数),所以**直接按这个 Agent 自己的可见性算**,没有特殊情况,没有兜底逻辑。

### 9.5 这套机制取代了什么

**取消了原来"数字员工之间必须显式配置关系才能互相发消息"的限制**。本期之后:

- 两个公开/定制 Agent -> 默认可以互通,不用配。
- 一个公开/定制 Agent + 一个私人 Agent -> 双方互不相通（公开调不到私人，私人也调不到公开）。
- 两个私人 Agent -> 仅同创建者的可以互通。
- **Agent 与人之间** -> 所有非私人 Agent 默认可以与公司所有人协作;私人 Agent 仅可与创建者协作。

这是一个明显的简化,但也是一个产品判断:之前显式关系起了"白名单"作用,现在改成"按可见性默认通信",失去了"两个公开 Agent 之间也能精细控制"的能力。本期判断小团队场景不需要这种精细度。

## 10. 第三方组织架构 tab

这个 tab 单独陈列,跟公司花名册并列。

### 10.1 视图形态

按**同步源**分子区域:

```Plaintext
飞书(同步于 2026-05-26 10:00)
[组织树,可展开收起,按飞书的部门结构呈现]
每个员工节点显示:姓名 / 飞书职位 / 是否已映射

Google Workspace
...
```

如果某个同步源还没接通,该子区域显示空状态 + "去企业设置开启同步"的跳转。

### 10.2 映射操作

每个第三方员工节点都有一个状态:

- **已映射**:已经关联到平台某个账号,卡片上显示对应平台员工的姓名。
- **未映射**:灰色 + "待映射"角标。

**映射操作**(仅管理员):

- 选中第三方员工 -> "映射到平台员工" -> 弹出搜索框,展示平台上**尚未被任何第三方员工映射过**的账号 -> 选定 -> 完成。
- 映射完成后:
  - 这个第三方员工节点状态变"已映射"。
  - 在公司花名册 tab 那边,对应的平台员工卡片上多一个"来自飞书"的徽标。
- **解除映射**:右键已映射节点 -> "解除映射" -> 确认 -> 状态变回"未映射",平台员工那边的徽标消失。**映射只是身份关联,不会因为解除映射而删除任何账号或数据**。

### 10.3 这个 tab 不做什么

- 不允许**在这个 tab 里编辑**第三方的组织架构(部门、汇报关系等)-- 那是第三方平台的事,要改去飞书改。
- 不做**反向同步**(把平台的部门推回飞书)。
- **不做"一键导入飞书部门到花名册"** -- 花名册的部门一律由管理员在主 tab 里自己定义。两边的部门是独立的概念,第三方的"销售部"和花名册的"销售部"不会自动对应。这是有意的产品边界:花名册是平台原生的组织,不被第三方组织架构覆盖。

## 11. 对旧逻辑的修改

本期不只是"新增一个花名册页面",还**完整替换了原来"关系"那套机制**。下面把要删/要改的全部列清楚,避免遗漏。

### 11.1 删除"关系"这个概念

- 数字员工详情页里的\*\*"关系" tab 整个删除\*\*。该 tab 原本承担两件事:配置该 Agent 跟人的关系(上下级/协作/导师/...)、配置该 Agent 跟其它 Agent 的关系(同事/上下级/...)。两件事在新版都不存在了。
- 数据库里现有的关系配置不再被任何地方使用,作为遗留数据处理(详见 §11.5 数据迁移)。
- 跟"关系"配套的概念(关系类型枚举、关系描述文本、关系网络可视化)全部从产品里消失。

### 11.2 "定制可见"简化为仅管理权限

- Agent 的"定制可见"档保留,但**使用权限统一按公开处理**,"定制"仅控制管理权限。
- 历史的"定制可见"Agent 中配置的使用权限自动失效,管理权限保持不变,升级自动完成。

### 11.3 替换"数字员工之间能否互通"的判定逻辑

| 旧逻辑 | 新逻辑 |
|-|-|
| Agent A 能否给 Agent B 发消息 = **A 跟 B 之间是否显式配过关系** | Agent A 能否给 Agent B 发消息 = **B 是否在 A 的可见性范围内**(由可见性规则静态决定,见 §9.2) |
| 关系数据要逐个手动配,NxM 次劳动 | 0 次配置,公开 Agent 默认全互通 |

这是本次最大的内核改动 -- **数字员工互相"认识"的来源,从"创建者显式声明"变成"花名册可见性自然决定"**。

### 11.4 替换数字员工"心里那本认识谁的清单"的来源

- 旧:每个数字员工有一份"我认识哪些同事"的预生成清单,关系一变要重新生成,容易过期。
- 新:每次数字员工要回应时,**实时从花名册算清单**,不存在过期问题。

(实现机制由研发评估,不在业务 PRD 范围内,但产品上的可感知差异是:**关系修改实时生效,没有"为什么我刚改了 Agent 还不知道"的延迟问题**。)

### 11.5 数据迁移

上线时:

- 全公司所有"关系"数据(人-Agent 关系、Agent-Agent 关系)**全部归档,不再被读取**,但暂不物理删除,保留 30 天观察期,确认没有问题后再清理。
- 所有公开 Agent **自动获得与其它公开 Agent 互通的能力**,无需创建者干预。

### 11.6 花名册查询工具

为数字员工提供一个新的内置工具 `query_roster`,让 Agent 能够主动查询公司花名册:

- **功能**:返回当前公司花名册中 Agent 可见范围内的所有成员列表(包含人和数字员工),每个成员附带:姓名/名称、角色描述、所属部门、类型(人/数字员工)。
- **用途**:Agent 通过此工具了解公司有哪些成员,然后自主决定找谁协作。这取代了原来的 `relation.md` 文件机制。
- **可见性规则**:工具返回的结果遵循 §9.2 的可见性规则--工具返回的结果遵循 §9.2 的可见性规则--公开/定制 Agent 能看到所有非私人成员（即公开/定制 Agent 和所有人类同事）;私人 Agent 仅能看到同一创建者的其它私人 Agent 和创建者本人。
- **与现有工具的关系**:Agent 先通过 `query_roster` 了解公司有哪些同事,再通过已有的 `send_message_to_agent`（联系数字员工）、`send_platform_message`（联系平台用户）或 `send_feishu_message`/`send_channel_message`（联系飞书/渠道用户）发起协作。
- **实时性**:每次调用实时查询,不存在缓存过期问题。

## 13. 待评审讨论的开放问题

本期主要产品决策已基本明确,以下问题留待评审时进一步对齐:

1. **公开 Agent 不能调自己创建者的私人 Agent -- 这个"严格隔离"会不会让创建者觉得别扭?** 比如 Tom 创建了一个公开的"周报助手"和一个私人的"个人备忘助手",Tom 自己跟周报助手对话时,周报助手也调不到他自己的备忘助手。Tom 可能期望"我自己的两个 Agent 之间应该能通"。是否需要为这种"同创建者"场景开一个例外?
2. **第三方组织架构 tab 的同步频率**:飞书/Google/Slack 的组织架构变更,需要多快反映到这个 tab?定时同步(每小时/每天)、手动触发、还是实时?

## 14. 不在本期范围

- 数字员工之间或人之间的显式"关系"(上下级 / 协作 / 导师等)- 本期明确不做。
- **人的卡片点击跳转员工档案详情页** -- 本期不做,人的卡片不可点击。
- **部门嵌套层级**(销售部 / 华东区销售组 / 华南区销售组) -- 本期只一层部门。
- **跨部门的同一个成员** -- 一个人/Agent 只能在一个部门里。
- **工号/成员编号** -- 本期只用姓名识别。
- **数字员工独立的"花名册简介"字段** -- 复用 Agent 现有的 role 字段。
- **一键把第三方组织架构导入到花名册的部门结构** -- 两边的部门是独立的概念。
- 花名册导出(PDF / Excel)。
- 数字员工能力的全文搜索(本期搜索只匹配姓名和 Agent 名)。
- 飞书/Google/Slack 同步逻辑本身。
- 把平台花名册反向推回第三方平台。
- 移动端的花名册形态(桌面优先)。
