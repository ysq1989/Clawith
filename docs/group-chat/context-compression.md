# 群上下文压缩

> 状态说明：本文保留方案调研和来源依据，不定义 Run 状态所有权。最终数据模型、预算与执行规则以 `chat-model-refactor.md`、`technical-design.md` 和 `docs/single-agent-runtime/technical-design.md` 为准。当前已确定统一使用 `chat_sessions`、`chat_messages`、`session_context_states`；LangGraph checkpoint 是 Run 执行状态唯一事实源；群共享压缩使用显式配置的独立 Compact 模型。

## 问题

群聊上下文会快速增长。如果每个被唤醒的 Agent 都收到完整群聊记录，提示词里会混入大量无关内容：闲聊、重复确认、过期状态、失败尝试、重复工具日志和已经失效的决策。

这是一个核心产品问题，也是核心架构问题。目标不只是减少上下文长度，而是为当前这一轮保留正确的工作上下文。

## 工作方向

群上下文应该分层加载，而不是直接注入完整群聊记录。

建议给被唤醒 Agent 组装以下上下文包：

1. 当前触发消息，包括谁 @ 了谁。
2. 最近原始消息窗口。
3. 相关群记忆。
4. 群工作区文件索引和被选中的文件引用。
5. 持续维护的群摘要或主题摘要。
6. 可选展开的历史记录：当 Agent 明确需要某段旧讨论、某个文件或某个决策时再加载。

## 主流方案总览

主流实践基本都不是单一方案，而是把“完整记录存储”和“模型可见上下文”分开：系统保留完整群聊记录、文件和审计日志，但每次唤醒 Agent 时只组装一个有预算、有优先级、可追溯的上下文包。

### 1. 最近窗口

做法：只把最近 N 条消息，或最近一段 token 预算内的原始消息放进上下文。

优点：

1. 实现简单，行为稳定。
2. 适合短对话和连续协作。
3. 可以保留最近语气、指代关系和还没沉淀的细节。

缺点：

1. 容易丢失较早的关键决策。
2. 群聊里如果有大量闲聊，最近窗口也会很脏。
3. 只解决长度问题，不解决内容相关性问题。

适合 Clawith 的用法：作为必备底座，但不能单独使用。建议只保留“最近相关消息窗口”，而不是无差别保留最近全部消息。

### 2. 消息裁剪和删除

做法：在进入模型前裁剪或删除部分消息，例如删除最早消息、删除失败工具结果、删除已经被摘要覆盖的旧过程。

优点：

1. 直接降低 token。
2. 对成本和延迟有明确收益。
3. 对工具调用密集的 Agent 工作流尤其有效。

缺点：

1. 删除不当会破坏模型需要的因果链。
2. 对带工具调用的消息，需要保留模型供应商要求的成对结构，例如工具调用和工具结果不能随意拆开。
3. 如果只删除不摘要，会丢失可恢复性。

适合 Clawith 的用法：可以用于“无价值过程”的清理，例如旧的失败尝试、过期搜索结果、重复工具日志。但需要保留完整原始日志作为审计源，模型上下文里的删除不等于数据删除。

### 3. 滚动摘要

做法：当群聊达到消息数或 token 阈值后，把较早的原始消息总结成摘要，后续上下文包含摘要和最近消息。

优点：

1. 成熟、常见，适合长对话。
2. 能保留跨轮次状态，例如决策、待办、责任人、文件引用。
3. 可以增量维护，不必每轮从完整群聊重新生成。

缺点：

1. 摘要是有损压缩，可能遗漏细节。
2. 摘要如果不可编辑，会把错误长期带入后续判断。
3. 群聊多主题混杂时，一份全局摘要会变脏。

适合 Clawith 的用法：第一版建议做结构化滚动摘要，但摘要只作为工作上下文，不作为不可挑战的事实源。关键事实进入群记忆前需要确认。

### 4. 服务端或 SDK 自动压缩

做法：由模型平台或 SDK 在上下文超过阈值后自动生成压缩项或摘要，并用压缩后的内容继续对话。OpenAI 和 Anthropic 都已经提供类似的 compaction/context management 能力。

优点：

1. 接入成本低。
2. 适合长时间 Agent 会话。
3. 能在接近上下文限制时自动维持连续性。

缺点：

1. 压缩结果可能是黑盒或不适合用户审阅。
2. 不一定理解 Clawith 的群、文件、权限、责任人等业务结构。
3. 自动压缩可能带来额外延迟。

适合 Clawith 的用法：可以作为底层模型调用保护，但不应替代产品层的群上下文构造器。Clawith 仍需要自己维护群摘要、群记忆、文件索引和审计日志。

### 5. 长期记忆

做法：把历史中长期有效的信息抽取出来，形成可检索、可更新的记忆，例如用户偏好、项目背景、协作规则、确定决策。

优点：

1. 能跨会话保留稳定信息。
2. 比把完整聊天记录塞进上下文更干净。
3. 适合“不同群加载不同群记忆”的模型。

缺点：

1. 自动写入容易污染记忆。
2. 需要展示、编辑、删除和审计能力。
3. 群记忆、个人记忆、Agent 记忆之间必须隔离。

适合 Clawith 的用法：群记忆应该作为单独层存在，像技能一样按群加载。第一版可以允许 Agent 建议写入，但实际写入建议由人确认。

### 6. 检索式历史

做法：把历史消息、文件、摘要、决策索引化。每轮根据当前消息、被 @ 对象、任务类型检索相关片段，而不是加载全部历史。

优点：

1. 适合长群聊和多主题群。
2. 可以按相关性召回旧决策、文件、讨论线。
3. 能和群工作区文件索引结合。

缺点：

1. 检索可能漏召回。
2. 只靠向量相似度会把语义相似但状态过期的内容召回。
3. 需要元数据过滤，例如时间、主题、文件、责任人、是否已废弃。

适合 Clawith 的用法：作为“按需展开历史”的主要机制。第一版可以先做关键词、时间、@ 对象、文件引用等结构化检索，再逐步引入向量检索。

### 7. 结构化状态账本

做法：把群里最重要的协作状态维护成结构化字段，而不是只靠自然语言摘要。例如目标、当前阶段、已确认决策、待办、责任人、文件、阻塞、废弃决策。

优点：

1. 比自然语言摘要更可控。
2. 方便 UI 展示和用户纠错。
3. 更适合群协作、项目推进和 Agent 接力。

缺点：

1. 需要定义字段和更新规则。
2. 对开放闲聊不一定适用。
3. 自动抽取仍然需要人审或置信度机制。

适合 Clawith 的用法：群摘要应优先做成“结构化状态 + 简短自然语言说明”，而不是一大段概括。

### 8. 分层上下文包

做法：每次 Agent 被唤醒时，由一个上下文构造器按预算和优先级组装内容。

建议优先级：

1. 当前触发消息和明确 @ 关系。
2. 群成员和被唤醒 Agent 的角色信息。
3. 最近相关原始消息。
4. 当前主题摘要。
5. 群记忆中被检索命中的条目。
6. 群工作区文件索引和必要文件片段。
7. 相关历史片段。
8. 审计或工具日志摘要。

优点：

1. 产品可控，适合 Clawith 自己的群、权限和工作区模型。
2. 可以按不同 Agent、不同群、不同任务组装不同上下文。
3. 可以解释“为什么 Agent 看到了这些内容”。

缺点：

1. 需要专门的上下文构造层。
2. 需要持续调参和观测。
3. 需要处理冲突、过期和权限问题。

适合 Clawith 的用法：这是第一版最应该采用的主方案。

### 9. 提示词缓存

做法：把稳定前缀缓存起来，例如系统提示词、Agent 定义、工具定义、群基础信息、稳定背景材料，降低重复处理成本。

优点：

1. 可以降低成本和首 token 延迟。
2. 适合群成员信息、Agent 角色、稳定群背景等重复内容。
3. 和分层上下文包可以互补。

缺点：

1. 缓存不等于压缩，不能减少模型实际需要理解的无关内容。
2. 动态内容频繁变化会破坏缓存命中。
3. 如果把脏上下文缓存起来，只是更便宜地重复脏上下文。

适合 Clawith 的用法：用于稳定层，不用于解决群聊内容污染问题。群成员列表、Agent 能力说明、群规则、固定工具说明可以考虑缓存。

### 10. 直接使用长上下文

做法：依赖大上下文模型，把更多群聊历史、文件和工具结果直接放进输入。

优点：

1. 接入最简单。
2. 短期内能减少工程复杂度。
3. 对少量长文档场景有帮助。

缺点：

1. 成本和延迟高。
2. 长上下文不等于高质量上下文；无关内容会干扰判断。
3. 相关信息在很长上下文中间时，模型可能更难稳定使用。

适合 Clawith 的用法：不能作为默认策略，只能作为 fallback 或特定高价值任务的扩展模式。

### 11. 多 Agent 轮次控制

做法：对群里 Agent 的发言、接力和停止条件做限制，例如只唤醒被 @ 的 Agent、一次只选一个下一发言者、最大消息数、最大轮次、超时、终止词。

优点：

1. 防止 Agent 互相 @ 导致无限接力。
2. 降低上下文膨胀速度。
3. 让上下文压缩和调度策略协同工作。

缺点：

1. 过强限制可能降低协作能力。
2. 智能选择下一 Agent 的 prompt 本身也会受长历史干扰。
3. 群内任务复杂时，需要更明确的任务拆解机制。

适合 Clawith 的用法：第一版应坚持显式 @ 唤醒，并设置最大轮次、最大 Agent 数、超时和停止条件。

## 方案成熟度判断

| 方案 | 成熟度 | 第一版建议 |
| --- | --- | --- |
| 最近窗口 | 高 | 必做，但只作为一层 |
| token 预算和裁剪 | 高 | 必做 |
| 结构化滚动摘要 | 高 | 必做 |
| 检索式历史 | 中高 | 建议做轻量版 |
| 群专属记忆 | 中高 | 建议做，但写入要谨慎 |
| 工具结果清理 | 中高 | 群工作区工具调用多后必做 |
| 服务端/SDK 自动压缩 | 中 | 可作为兜底，不作为产品主逻辑 |
| 提示词缓存 | 高 | 用于降本，不用于解决语义污染 |
| 直接长上下文 | 中 | 只做扩展模式或兜底 |
| 全自动记忆写入 | 低到中 | 第一版不建议默认开启 |
| 全量群 Agent 自动路由 | 低到中 | 第一版不建议默认开启 |

## 对 Clawith 的推荐组合

第一版建议采用“原始记录完整保存，模型上下文分层组装”的组合方案：

1. 完整群消息流永久保存，作为审计和回溯源。
2. Agent 被 @ 后，先构造本轮上下文包，而不是直接读取完整群聊。
3. 上下文包必须包含当前触发消息、被 @ 对象、群成员信息和最近相关消息。
4. 较早历史通过结构化群摘要进入上下文。
5. 长期稳定信息通过群记忆进入上下文。
6. 文件只默认注入群工作区索引和相关文件引用，完整文件内容按需读取。
7. 工具日志默认不完整注入，只保留结果摘要、错误摘要和审计引用。
8. Agent 可以请求展开某段历史、某个文件、某个决策来源。
9. 摘要更新和记忆建议尽量异步做，不阻塞用户看到回复。
10. 群记忆写入需要显式规则，第一版建议由人确认。

### 建议的第一版上下文包结构

```text
本轮输入
- 当前触发消息
- 被 @ 的人和 Agent
- 本轮任务类型

群基础信息
- 群名称
- 群成员列表
- 群内 Agent 列表及角色说明
- 群权限和可见范围

近期上下文
- 最近相关消息窗口
- 最近未完成工具调用或任务状态

长期上下文
- 命中的群记忆
- 当前主题摘要
- 已确认决策
- 未决问题

工作区上下文
- 群工作区文件索引
- 本轮命中的文件引用
- 必要时展开的文件片段

可追溯信息
- 摘要来源消息范围
- 关键决策来源消息
- 工具调用和文件变更审计引用
```

### 建议的第一版压缩流程

1. 每条群消息进入完整消息流，不立刻写入群记忆。
2. 达到消息数或 token 阈值后，后台生成或更新主题摘要。
3. 摘要更新时抽取：决策、待办、责任人、文件、阻塞、过期状态。
4. 如果摘要发现长期事实或协作规则，生成“建议写入群记忆”的候选项。
5. 用户确认后，候选项才进入群记忆。
6. Agent 被唤醒时，实时检索群记忆、主题摘要、近期消息和文件索引，组装上下文包。
7. 如果 Agent 判断上下文不足，可以请求展开历史或文件。
8. 每次展开都留下审计引用，避免“模型看过但用户不知道”的隐性上下文。

## 调研记录

### 1. 短期历史必须显式管理

LangGraph 的记忆文档列出了长对话的常见处理方式：裁剪消息、删除消息、总结早期消息、管理检查点，以及自定义过滤。

对 Clawith 的启发：群聊不应该默认把完整原始聊天记录作为提示词输入。

来源：https://docs.langchain.com/oss/python/langgraph/add-memory

### 2. 只裁剪近期消息不够

只保留最新消息成本低、行为确定，但会丢掉重要的历史决策。它只能作为其中一层，也就是“最近原始消息窗口”。

对 Clawith 的启发：需要保留最近原始消息窗口，但必须同时配合摘要和群记忆。

来源：https://docs.langchain.com/oss/python/langgraph/add-memory

### 3. 摘要应该增量维护

LangGraph 描述的做法是：生成摘要时把已有摘要作为上下文，然后删除更早的原始消息，只保留摘要和近期消息。

对 Clawith 的启发：群摘要应该在达到阈值后增量更新，而不是每次都从完整聊天记录重新生成。

来源：https://docs.langchain.com/oss/python/langgraph/add-memory

### 4. 会话输入应该支持自定义过滤

OpenAI Agents SDK 的会话能力支持通过回调控制历史记录和新输入如何合并，也支持限制取回的历史记录数量。

对 Clawith 的启发：群上下文构造器应该成为独立层，在每次调用 Agent 前负责过滤、排序、限量和组装上下文。

来源：https://openai.github.io/openai-agents-python/sessions/

### 5. 自动压缩有延迟成本

OpenAI Agents SDK 提到，自动压缩可能阻塞流式响应；如果低延迟很重要，建议在轮次之间或空闲时间做压缩。

对 Clawith 的启发：群摘要应尽量异步维护，或在空闲窗口维护，不要在每次可见回复时同步生成。

来源：https://openai.github.io/openai-agents-python/sessions/

### 6. 长期记忆通常会分成静态信息、事实抽取和向量检索

LlamaIndex 的记忆设计里，短期记忆会按 token 预算保留最近消息；长期记忆可以由静态记忆、事实抽取记忆、向量记忆等模块组成，并通过优先级控制超预算时哪些内容先被移除。

对 Clawith 的启发：群记忆不应该是一坨文本，而应该至少区分“稳定群信息”“已确认事实/决策”“可检索历史片段”。不同类型应该有不同优先级和不同写入规则。

来源：https://developers.llamaindex.ai/python/framework/module_guides/deploying/agents/memory/

### 7. 服务端压缩正在成为平台级能力

OpenAI 的 compaction 文档描述了服务端压缩：当渲染后的 token 数超过阈值时，服务端运行压缩，把关键状态和推理带到下一轮，同时减少 token。Anthropic 也把 server-side compaction 作为长时间对话和 Agent 工作流的主要上下文管理策略之一。

对 Clawith 的启发：平台级压缩可以作为兜底能力，但它不理解 Clawith 的群记忆、群工作区、权限和审计边界。因此产品层仍然需要自己的上下文构造策略。

来源：https://developers.openai.com/api/docs/guides/compaction

来源：https://platform.claude.com/docs/en/build-with-claude/context-windows

### 8. 上下文编辑适合清理工具结果和推理过程

Anthropic 的 context editing 支持清理旧工具结果和 thinking blocks，并强调上下文是有限资源，无关内容会降低模型关注度。旧工具结果、搜索结果、文件内容一旦被处理完，通常不需要长期留在活跃上下文里。

对 Clawith 的启发：群工作区场景里，工具结果要分成“最终结果”“可审计原文”“模型可见摘要”。模型上下文里保留最终结果和引用即可，完整工具输出进审计日志或文件，不要长期挤占上下文。

来源：https://platform.claude.com/docs/en/build-with-claude/context-editing

### 9. 提示词缓存是降本方案，不是压缩方案

Anthropic 和 OpenAI 都提供 prompt caching。缓存适合稳定前缀，例如系统提示、工具定义、背景资料、长多轮对话中的重复前缀，可以降低重复处理成本和延迟。

对 Clawith 的启发：缓存不能解决“群聊里无用内容太多”的质量问题。它只能让稳定内容更便宜、更快。Clawith 可以缓存群规则、Agent 定义、工具定义、稳定群背景，但仍然需要过滤动态群消息。

来源：https://platform.claude.com/docs/en/build-with-claude/prompt-caching

来源：https://developers.openai.com/api/docs/guides/prompt-caching

### 10. 长上下文本身不是答案

Anthropic 的 context window 文档明确提到，更多上下文不自动等于更好；token 增长会带来准确率和召回下降，也就是 context rot。经典研究 Lost in the Middle 也指出，相关信息出现在长上下文中间时，模型表现可能显著下降。

对 Clawith 的启发：不能因为模型支持长上下文，就把完整群聊都塞进去。群聊更需要“上下文工程”：把高价值内容放在稳定、靠前、结构化的位置，把低价值内容放到可检索历史里。

来源：https://platform.claude.com/docs/en/build-with-claude/context-windows

来源：https://arxiv.org/abs/2307.03172

### 11. 检索和摘要结合是长期对话里的成熟方向

Beyond Goldfish Memory 的长期开放域对话研究显示，检索增强方法，以及具备总结和回忆过往对话能力的方法，在长期对话场景中优于只依赖短上下文的标准模型。

对 Clawith 的启发：群聊历史应该同时支持“摘要态”和“检索态”。摘要负责连续协作状态，检索负责找回具体旧消息、文件、决策来源。

来源：https://arxiv.org/abs/2107.07567

### 12. 虚拟上下文管理强调多层记忆

MemGPT 提出 virtual context management，把有限上下文看作快速内存，把外部存储看作慢速内存，通过在不同记忆层之间移动信息，让 Agent 能在有限上下文窗口内处理更长历史。

对 Clawith 的启发：群聊可以采用类似分层：活跃上下文、群摘要、群记忆、群工作区、完整历史。Agent 不需要一次看到全部，但要能按需取回。

来源：https://arxiv.org/abs/2310.08560

### 13. 多 Agent 群聊需要终止限制

AutoGen 的 SelectorGroupChat 使用共享上下文和发言者选择机制，但示例里也包含终止条件，例如特定终止文本和最大消息数限制，用来避免无限循环。

对 Clawith 的启发：上下文压缩和轮次控制是关联问题。系统需要最大轮次、最大 Agent 数、超时时间和停止条件。

来源：https://microsoft.github.io/autogen/stable/user-guide/agentchat-user-guide/selector-group-chat.html

## 第一版建议策略

第一版建议使用保守的上下文包：

1. 始终包含当前消息。
2. 始终包含群成员列表。
3. 包含最近 N 条相关消息，不一定包含每条原始消息。
4. 包含由当前消息和被 @ 对象检索出的群记忆。
5. 对更早的上下文包含一份紧凑的持续摘要。
6. 包含群工作区文件索引，但默认不包含完整文件内容。
7. 允许 Agent 显式请求某个文件或某段历史讨论。
8. 不自动把群聊记录写入群记忆。
9. 在消息数量或上下文长度达到阈值后异步更新摘要。

## 摘要结构

群摘要应该保留结构化状态，而不只是自然语言概括。

建议字段：

1. 当前目标或主题。
2. 已确认决策。
3. 未决问题。
4. 当前任务状态。
5. 负责人和责任分工。
6. 重要文件引用。
7. 最近已完成事项。
8. 已知阻塞。
9. 已被替代或已经过期的决策。

## 已收敛与后续问题

已收敛：最近窗口默认 20 条；每个 `chat_session` 一份当前 Session Context；硬压缩阈值取群内有效 Agent 模型最小有效输入预算的 85%；共享压缩使用独立 Compact 模型；Agent 工具过程不作为公开群上下文事实。

后续产品增强不阻塞 v1：用户是否可编辑摘要、摘要更新是否在群时间线可见，以及文件重命名或删除后的长期引用治理。
