# 一、产品概述

Clawith 群聊 v1 是一个支持多人和多 Agent 同处一群的轻量协作群。它不是一个新的任务系统，而是让用户在熟悉的群聊形态里，把人和 Agent 拉到同一个协作空间中工作。

| 范围 | 说明 |
|-|-|
| **群聊 v1 重点** | 跑通人在群里邀请人和 Agent、创建和使用群 session、通过 @ 唤醒 Agent、让 Agent 基于当前群 session 的上下文和群 workspace 参与协作的核心链路。 |
| **后续规划** | Agent 主动邀请成员、智能路由、全量唤醒、定时任务、长期信息沉淀、更完整的任务化协作和更细的权限治理。 |

# 二、核心功能

## 2.1 名词定义

| 概念 | 定义 |
|-|-|
| 群 | ~~人和 Agent 共同参与的长期协作空间。~~人和Agent共同参与的群聊。 |
| 群成员 | 群里的参与对象，包括人和 Agent。 |
| 群公告 | 群级别的 Markdown 文档，用于记录群目标、协作规则和 Agent 在该群内的行为要求。 |
| 群 session | 群里的一个独立会话或话题。 |

## 2.2 创建群与邀请成员

用户可以创建一个群，建群人默认成为群管理。

### 2.2.1 群基本信息

群基本信息包括群名称、群介绍和群公告。

- 群名称用于识别这个群。
- 群介绍用于说明这个群的用途、背景或协作范围。
- 群公告用于记录群目标、协作规则和 Agent 在该群内的行为要求。
- 群内所有人类成员都可以查看、修改群名称、群介绍和群公告。
- Agent 不能修改群名称、群介绍或群公告。
- 群管理可以邀请其他人和 Agent 入群，也可以将成员移出群。
- 群管理可以解散一个群，解散后，群内的所有数据，例如session，群workspace文件等都会被硬删除。
- 普通成员可以邀请其他人和 Agent 入群，但不能将成员移出群。
- 邀请成员时，备选范围是邀请人可见的人和 Agent。
- 用户能私信的 Agent，默认可以被邀请进群。
- Private Agent不允许被邀请进群。
- Agent 可以访问群成员列表，了解群里有哪些人和 Agent，并在任务卡住时 @ 合适的群成员寻求帮助。

### 2.2.2 邀请成员

群成员包括人类成员和 Agent。

- 建群人默认成为群管理。
- 群管理可以邀请其他人和 Agent 入群，也可以将成员移出群。
- 普通成员可以邀请其他人和 Agent 入群，但不能将成员移出群。
- 邀请成员时，备选范围是邀请人可见的人和 Agent。
- Private Agent 不允许被邀请进群。
- Agent 不能主动邀请新成员入群，也不能将成员移出群。

### 2.2.3 第三方同步成员

第三方同步成员指从飞书、钉钉、企业微信、Google Workspace 等外部组织通讯录同步进来的组织成员。

- 第三方同步成员可以出现在邀请成员的候选列表中。
- 已绑定平台账号的第三方同步成员，入群后按普通人类成员处理。
- 未绑定平台账号的第三方同步成员，暂时不允许入群。

### 2.2.4 第三方群

- Clawith 原生群消息暂时不会自动同步到第三方私信或第三方群聊。
- 如果 Agent 需要主动联系第三方同步成员且未绑定平台账号，必须显式调用渠道发送能力，例如 `send_channel_message`，发送一条私信，目标是具体成员。

## 2.3 群公告

群公告是群级别的 Markdown 文档，用于记录群目标、协作规则和 Agent 在该群内的行为要求。群公告应保存稳定、简洁的群级规则和协作背景，不用于承载长文档、完整历史或大量过程内容。

- 群公告属于群，不属于某个群 session，也不属于某个人类成员或Agent。
- 群成员可以查看群公告。
- 人类群成员可以编辑群公告。
- Agent 不允许编辑群公告。
- 群公告的长度不设置上限，但是不推荐用户写过长的内容。

### 2.3.1 群公告注入机制

- Agent 在群内被 @ 时，群公告会作为prompt进入上下文。
- 群公告修改后，在下一次注入 Agent 上下文时生效，不影响已经开始执行的 Agent 回复或任务。
- 群公告进入 Agent 上下文时有长度上限。
- 超过上限的部分不会自动进入本轮上下文。
- Agent 不会因为群公告被截断而自动继续读取剩余内容。
- 如果 Agent 需要查看更多群公告内容，需要通过后续的群公告读取工具搜索并读取。

## 2.4 群 session

群 session 是群里的独立消息流，用于承载一个主要讨论目标或一段连续协作。

### 2.4.1 群 session 基本规则

- 群创建后，不自动创建群 session。
- 群 session 只允许成员手动创建。
- 群成员进入一个没有 session 的群时，界面展示空状态，并引导创建群 session。
- 群 session 属于群，不属于某个成员或某个 Agent。
- 同一个群下可以有多个群 session。
- 群成员可以查看群内 session 列表，并进入未删除的群 session 查看消息。

### 2.4.2 创建群 session

- 人类群成员可以在群内创建新的群 session。
- Agent 不允许创建新的群 session。

### 2.4.3 群 session 命名

- 新建群 session 时，系统先生成临时名称。
- 群 session 产生第一条消息后，如果仍是临时名称，系统可以用第一条消息内容生成标题。
- 人类群成员可以手动修改群 session 名称。
- Agent 不允许修改群 session 名称。

### 2.4.4 删除群 session

- 只有群管理可以删除群 session。
- 删除群 session 后，该 session 不再出现在默认 session 列表中。
- 删除群 session 后，Agent 后续不会再基于该 session 的历史消息组装上下文。
- 删除群 session 不会删除群本身。
- 删除群 session 不会删除群公告。
- 删除群 session 不会删除群成员。
- 删除群 session 不会删除群 workspace 中的文件或产物。
- 删除群 session 后，不提供用户侧恢复能力。
- primary session 也允许删除；如果仍有其他未删除 session，系统自动选择最近活跃的一个作为新 primary。
- 如果已经没有其他 session，群可以暂时没有 primary；后续由人类成员创建的第一个 session 自动成为 primary。
- 删除 session 时，依赖该 session 的前台对话、Planning 和协作 Run 取消；独立 Task、Trigger、Heartbeat 等后台业务继续执行。
- 后台业务返回结果时优先写回原 session；原 session 已确认不可用时，可以回退到同一群当前 primary。没有可用 primary 时只记录交付失败，不自动创建群 session。

### 2.4.5 删除群

- 只有群管理可以删除群。
- 删除群后，该群不再出现在群列表中。
- 删除群后，群 session、群公告和群 workspace 不再对用户可见。
- 删除群后，不提供用户侧恢复能力。
- 系统可以保留必要的后台归档数据，用于排障、审计或合规留存。
- 后台归档不等同于用户可恢复能力。

### 2.4.6 群 session 的上下文边界

- 群 session 用于隔离消息上下文，不用于隔离群 workspace。
- 不同群 session 的消息默认不互相进入上下文。
- 同一个群下的多个群 session 共享同一个群 workspace。
- Agent 被 @ 后，默认基于当前群 session 组装上下文。

## 2.6 群消息与 @ 唤醒

### 2.6.1 群消息

群消息是人类用户或 Agent 在某个群 session 中公开发送的消息。

- 人类群成员可以在群 session 中发送消息，并在消息任意位置 @ 一个或多个 Agent。
- Agent 被 @ 后，在当前群 session 中回复消息。
- Agent 最终发送到群 session 里的内容属于群消息。
- Agent 的中间工作过程是异步执行过程，不发送到群中，也不作为群消息。

### 2.6.2 @ 唤醒规则

- 人类群成员 @ 人不会触发 Agent。
- 人类群成员可以在一条消息中 @ 一个或多个 Agent。
- 被 @ 的 Agent Run 已可靠落库并被系统接受处理后，需要在当前群 session 中发送一条普通群消息，确认自己已被唤醒。
- 唤醒确认只使用普通群消息，不提供动画或独立状态；后续处理结果继续通过普通群消息回复。
- 只要已经发送唤醒确认，后续必须通过普通群消息给出用户可见结果：成功回复、需要用户补充信息、处理失败或已取消，不能只在内部日志结束。
- 失败消息只说明可公开的失败原因和下一步建议，不展示堆栈、工具参数、凭据或内部调试信息。
- 未被 @ 的 Agent 不处理当前消息。

### 2.6.3 Agent 回复规则

- Agent 被 @ 后，只在当前群 session 中回复消息。
- Agent 可以在回复中 @ 一个或多个群内已有人类成员和 Agent。
- 在群内对话场景中，Agent 没有需要人审批或需要人帮助时，不允许主动发送群消息。

### 2.6.4 未唤醒 Agent 的行为边界

- 没有被 @ 的 Agent 不会处理消息。
- 没有被 @ 的 Agent 不会因为群内消息、群公告或成员变化而更新自己的群 memory。

### 2.6.5 任务触发与回调消息

- 如果人类群成员在群内为某个 Agent 创建了触发器或需要回调的任务，该 Agent 可以在任务触发或任务完成时主动发送群消息。
- 任务触发或回调产生的群消息，应发送到对应的群 session。
- 任务触发或回调产生的群消息，如果涉及到 @ 其他成员和Agent的情况，按照 @ 唤醒规则处理。

### 2.6.6 未读消息规则

未读状态用于帮助人类群成员判断哪些群 session 有新的公开群消息。未读状态按“成员 + 群 session”维度记录，不按整个群统一记录。

- 每个人类群成员在每个群 session 中都有独立的未读状态。
- 用户进入某个群 session 并看到最新消息后，该用户在该 session 的未读状态更新为已读。
- 用户自己发送的群消息，不计入自己的未读。
- 其他人类成员发送的群消息，计入未读。
- Agent 最终发送到群 session 的回复，计入未读。
- 任务触发或回调产生的群消息，计入未读。
- Agent 的中间工作过程不作为群消息，也不计入未读。
- 群 workspace 文件创建、修改或删除本身不直接计入未读；如果需要提醒成员，应由对应的群消息承载。
- 群公告修改不直接计入群 session 未读；如需要通知成员，应由群消息或后续通知机制承载。
- 被 @ 的人类成员可以在未读之外展示 @ 提醒。
- 删除群 session 后，该 session 对应的未读状态不再展示。
- 删除群后，该群下所有未读状态不再展示。
- 同一成员的已读位置只能向更新的消息推进；多端或延迟请求不能把已读位置回退。

## 2.7 多 Agent 任务规划与群内 A2A



同一条消息同时 @ 多个 Agent 时，系统进入任务规划阶段。只 @ 一个 Agent 时，不触发任务规划 Agent，按单 Agent 唤醒规则处理。如果用户在消息中已经明确给出分工、顺序或协作方式，任务规划 Agent 应以用户给出的规划为准。

### 2.7.1 任务规划 Agent

- 任务规划阶段由系统内置的任务规划 Agent 负责。
- 任务规划 Agent 不作为普通群成员展示，默认不在群里发言。
- 任务规划 Agent 只负责生成本次协作的轻量分工计划，不替代业务 Agent 完成具体专业工作。
- 任务规划过程默认是系统内部过程，不作为群消息展示。
- Planning 使用单独配置的模型，不复用任意群成员 Agent 的模型，也不复用 Compact 模型。
- Planning 模型未配置、配置无效或最终调用失败时，系统必须在当前群 session 中发送显式失败提示，不得静默跳过规划或默认把全部任务并发执行。

### 2.7.2 轻量分工计划

轻量分工计划用于给被 @ 的多个 Agent 提供初始协作方向，并必须输出本轮执行策略。执行策略只约束 Run 的启动顺序和依赖关系，不限制 Agent 在各自步骤内的自主判断。

轻量分工计划需要明确：

- 本次协作目标。
- 被 @ 的 Agent 列表。
- 每个被 @ Agent 的建议分工。
- 执行策略：`parallel`、`sequential` 或 `dependency`。
- 每个步骤依赖的前置步骤；没有依赖时为空。

### 2.7.3 执行方式

- 用户明确给出分工、顺序或依赖时，执行策略必须遵循用户要求。
- 用户没有明确要求且步骤之间不存在结果依赖时，默认使用 `parallel`。
- 存在明确线性顺序时使用 `sequential`；存在部分步骤可并发、部分步骤依赖前置结果时使用 `dependency`。
- 系统只启动所有前置依赖已经完成的步骤，不得提前启动仍依赖其他结果的 Agent Run。
- 实际调度以步骤的前置依赖为准：`parallel` 的所有步骤都没有前置依赖；`sequential` 中每个后续步骤依赖前一步；`dependency` 表达任意无环依赖图。
- 执行策略与步骤依赖不一致时，本次规划无效，系统重新规划或返回规划失败，不自行猜测执行顺序。
- 被 @ 的 Agent 在自己的分工范围内自主判断如何完成任务。
- 被 @ 的 Agent 的回复写回当前群 session。

### 2.7.4 群内 A2A

群内A2A的发起和普通的A2A相同，消息和结论都不放到群内过程中。

Agent 自动 @Agent 或发起 A2A 时只限制循环，不限制正常的连续协作链。系统沿当前 Run 的父链检查 Agent 之间的有向跳转；同一条有向跳转每重复出现一次记为一次循环，例如 `A → B → A → B` 中第二次 `A → B` 记为一次。正常的 `A → B → C` 不计数并可以继续。当前链累计达到 5 次循环时，停止继续自动唤醒 Agent，并由当前 Agent 告知用户需要人工决定是否继续。

## 2.8 群 memory

群 memory 是 Agent 针对某个群维护的独立 memory.md，用于记录该 Agent 在这个群里的长期记忆。

- 归属

  - 群 memory 属于“Agent + 群”的组合。
  - 保存在群workspace中
  - 同一个 Agent 在不同群里有不同的群 memory。
  - 不同 Agent 在同一个群里也有各自独立的群 memory。
- 加载

  - Agent 在群内被 @ 时，加载该 Agent 对应这个群的群 memory。
  - Agent 在私信中被唤醒时，不加载群 memory。
  - Agent 在其他群中被唤醒时，不加载当前群的群 memory。
- 读，写入和删除

  - Agent 可以读取所有群内 Agent 针对这个群的群 memory。
  - Agent可以写入自己的群memory，不能修改其他Agent的群memory。
  - 人类用户可以读，写入和删除所有的群memory。
  - Agent 在群内完成一次被 @ 触发的处理后，可以判断是否需要更新该 Agent 在该群的群 memory。如果需要更新，Agent 在本次回复或任务结束后发起群 memory 更新。
  - 如果群被删除，对应的群memory也会同时被删除
- v1 中，Agent 自身 memory 不会自动从群 memory 中学习内容。



## 2.9 群上下文

Agent 被 @ 唤醒时，不直接加载完整群聊记录，而是由系统基于当前群 session 组装本轮上下文。最终给大模型的输入由 Agent 自身上下文和本轮群上下文组成。

Agent 自身上下文复用单 Agent 对话中已有的上下文机制，包括 `soul.md`、Agent 自身 memory、skills、tools、Agent workspace 能力说明等。群聊不会为 Agent 自身上下文重新定义一套新机制。

本轮群上下文包括，具体拼装机制，细则会放到技术文档中：

1. 当前触发消息全文。
2. 本轮 @ 关系，包括谁发送了消息、消息里 @ 了哪些人和 Agent、本轮实际唤醒的是哪个 Agent。

4. 当前 session 最近 20 条消息。
5. 当前 session 的结构化 Session Context，包括摘要、已确认要求、决策、未完成事项和引用。
6. 群公告全文。
7. 当前 Agent 在该群的群 memory 全文。
8. 当前 Agent 对该历史摘要的工作视角或已承担事项。
9. 群 workspace 中与本轮相关的流程记录、共享上下文资产、产物索引、文件摘要或必要片段。

- 本轮执行要求、工具权限、工具调用结果摘要和审计摘要属于系统指令或运行约束，不作为群上下文内容项单独列出。
- 群成员信息不默认全量进入 Agent 上下文。Agent 默认只获得当前发言人、被 @ 对象和必要的群基础信息；当 Agent 需要了解群内成员、寻找合适的人或 Agent、或在回复中 @ 某个成员时，应通过群成员查询工具按需获取。
- 群成员查询通过工具完成，工具只返回当前群内成员，支持按姓名、角色或能力检索，并返回可用于 @ 的稳定成员 ID。Agent 不应凭历史上下文中的名字猜测成员身份；但可以基于当前上下文判断在遇到问题时应向哪些群成员寻求协助，或应在回复中通知哪些群成员。

### 2.9.1 历史摘要与压缩机制

群聊与普通聊天复用统一 `chat_sessions`、`chat_messages` 和 Session Context 机制。当前群 session 的上下文由结构化 Session Context、待压缩区和最近消息组成；群聊不维护另一套群专用消息表或摘要事实源。

- 最近 20 条消息始终保留原文进入上下文。
- 滑出最近 20 条的旧消息先进入待压缩区。
- 待压缩区不会立刻压缩。
- 只有当待压缩区累计到系统设定的 token 规模时，才触发一次压缩。
- 压缩后，待压缩区清空，历史摘要更新。
- 如果 session 很短，可能完全不触发压缩。
- 如果 session 消息很多，也应按批次压缩，而不是每条消息都压缩。

历史摘要采用增量维护方式。系统更新历史摘要时，基于已有历史摘要和待压缩区中的旧消息生成新的历史摘要，避免每次回复都重新压缩完整历史。

群聊属于多 Agent 场景，必须显式配置独立 Compact 模型。该模型只负责生成共享 Session Context，不从群内 Agent 中临时选择；压缩阈值按群内所有有效 Agent 当前模型中最小的有效输入预算的 85% 计算，保证共享上下文可以被任一群内 Agent 使用。具体 Agent 被 @ 后的单次 Run 仍使用该 Agent 当前执行模型计算自己的 Runtime Context。



## 2.10 群 workspace

群 workspace 是群级别的工作区，用于沉淀群内协作过程中产生的文件、产物、流程和可复用上下文资产。群 workspace 的重点是保存后续群协作可能继续使用的内容。

群 workspace 中主要沉淀以下内容：

- 用户在群中上传或明确分享到群里的文件和资料。
- Agent 在被 @ 后执行任务时主动放入群 workspace 的文件，包括最终产物、阶段性结果、过程文件和整理材料。
- 人类成员要求 Agent 从 Agent 自己的 workspace 中复制到群 workspace 的指定文件或内容。
- ~~任务完成或阶段结束后沉淀的可复用工作内容，包括工作流程、关键结论、决策、过程偏好和后续类似任务可复用的经验。这类沉淀内容主要由系统的上下文压缩流程生成，群内 Agent 也可以在被 @ 后主动补充。~~

群 workspace 的基础规则：

- Agent 写入群 workspace 可以由用户明确要求触发，也可以由 Agent 在执行任务时自行判断。Agent 可以将与当前任务相关、后续可能复用的最终产物、阶段性结果、过程文件或整理材料写入群 workspace。
- 任务完成或阶段结束后的可复用工作内容，由 topic 状态判断触发。当系统判断当前 topic 已完成或进入阶段结束状态时，由上下文压缩流程生成工作流程、关键结论、决策、过程偏好等沉淀内容。
- 同群 Agent 被唤醒后，可以读取或搜索群 workspace 中的内容。
- 群 workspace 中的内容按目录组织，不采用纯平铺方式。
- v1 中，群 workspace 默认对群内成员开放基础管理能力。群内人类成员和群内 Agent 都可以创建、读取、修改、复制和删除群 workspace 中的内容。后续如需要更细的权限控制，再引入管理员、创建者、只读成员等角色规则。
- 跨空间内容采用复制方式，不做自动同步或版本联动。

### 2.10.1 跨空间内容复制

同一个 Agent 可以同时参与私信和群聊，也可以拥有自己的 Agent workspace，但不同空间的内容默认不互通。

<callout emoji="❗">
群聊 v1 允许用户显式要求 Agent 在不同空间之间复制指定内容，但 Agent 不能自主判断并主动搬运内容。所有跨空间复制都需要权限校验、发送前预览或确认、来源标记和审计记录。
</callout>

Agent workspace 和群 workspace 之间可以传递明确指定的文件或内容，但不能自动互通。

| 复制方向 | 群聊 v1 默认处理 |
|-|-|
| 私信 → 群 | 用户可以在私信中明确要求 Agent 将指定消息、结论或文件发送到指定群。发送后，该内容成为目标群的群消息或群 workspace 内容，后续可以进入对应群 session 的上下文或摘要候选。 |
| Agent workspace → 群 workspace | 用户可以在私信或群里明确要求 Agent 将 Agent workspace 中的指定文件、产物或整理结果分享到指定群。分享后，指定内容进入群 workspace，成为该群可复用内容。 |
| 群 workspace → Agent workspace | 用户可以在群里明确要求把群 workspace 中的指定文件、结论或材料交给某个 Agent 继续处理。复制后，指定内容进入该 Agent workspace，作为该 Agent 后续任务输入。 |
| 群 → 私信 | 用户在群里明确要求 Agent 将指定群消息、结论或文件发送给某个人时，Agent 可以以私信方式发送。私信接收方只能看到被显式发送的内容，不自动获得群历史或群 workspace 访问权。 |

统一规则：

- 必须由人显式触发，Agent 不能自主搬运私信、群消息或 workspace 内容。
- v1 只支持复制用户明确指定的文件或内容，不支持复制整个 Agent workspace、整个群 workspace 或完整群历史。
- v1 不支持空间之间的自动同步或版本联动。
- 复制前应展示将要发送或复制的内容；内容较长、包含文件或可能包含敏感信息时，需要用户确认。
- 只能复制触发人有权转发、复制或分享的内容。私有文件、他人消息、第三方渠道内容不能因为 Agent 看过就默认跨空间流转。
- 跨空间复制后，目标空间只获得被显式复制的内容，不继承源空间的完整上下文、历史记录或访问权限。



# 三、边界与后续规划

## 3.1 后续规划：定时执行

定时执行属于后续规划，不作为群聊 v1 需要专门完成的能力。v1 不为群聊单独建设新的定时任务系统。

- 现有触发器、回调任务或定时能力可以在群场景中复用。
- 定时或回调结果应写回对应的群 session。
- 复杂的一次性提醒、周期任务、群内计划任务管理和失败通知策略，后续再设计。

## 3.3 已确认技术边界

- 群领域使用 `groups`、`group_members`；群 session 和群消息复用统一 `chat_sessions`、`chat_messages`。
- 群内 Agent @Agent 复用统一 A2A / Agent Runtime，不把未公开过程写成群消息。
- Agent Run 的执行生命周期以 LangGraph checkpoint 为唯一事实源；群聊产品层只提交可靠 Runtime Command，并读取统一的可重建状态投影。
- Agent 内部执行日志、工具过程和 trace 不进入公开 `chat_messages`；只有公开回复写回当前群 session。
- 原生群聊和外部渠道群聊复用统一聊天模型，渠道差异由 adapter、`source_channel` 和 `external_conv_id` 表达。
- 群 Session Context 复用 `session_context_states`，共享压缩使用独立 Compact 模型；上下文装配、token 预算和压缩细则以技术方案为准。



## 3.4 审计方案

- v1 复用现有 `audit_logs`，不新增群聊专用审计表。
- 群创建、成员邀请或移除、管理员变更、Session 删除、群解散、公告修改和跨空间显式复制等管理操作需要记录操作者、租户、目标对象、动作、结果和时间。
- Agent Run 生命周期由统一 Runtime 的 checkpoint 投影与 `agent_run_events` 承载；工具详细过程进入 trace，不重复写入群聊审计日志。
- 审计记录不保存完整群消息正文、完整文件内容、模型 Prompt、凭据或未脱敏工具参数，也不提供用户侧恢复能力。

## 3.5 同一 Agent 被多条群消息 @ 时的处理

同一个 Agent 同时被多个人 @ 时，v1 按群消息写入顺序串行执行。这里的“顺序”以服务端成功写入群消息的顺序为准，不以客户端发送时间或前端展示时间为准。

该规则只约束同一个 Agent 的群 mention 队列，不表示 Direct Chat、Task、Trigger、Heartbeat 等所有 Run 都必须全局串行。

- 如果该 Agent 当前没有正在处理的消息，系统立即调用该 Agent。
- 如果该 Agent 正在处理上一条 @，新的 @ 等待上一条处理完成后再执行。
- v1 不并发执行同一个 Agent 的多个 @ 请求。
